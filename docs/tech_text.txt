Cache Schema Design (Redis)PharmaSys uses Redis 7+ as the primary caching layer. The cache is organised into functional namespaces with explicit TTL policies and invalidation strategies. Redis Cluster mode is recommended for production deployments with 3 primary + 3 replica nodes.2.1 Cache Architecture PrinciplesCache-Aside pattern: application reads cache first; on miss, fetches from DB and populates cacheWrite-Through for critical entities (stock levels, user sessions)Write-Behind (async) for non-critical counters (ranking scores, view counts)All cache keys follow a hierarchical namespace: pharmasys:{env}:{module}:{subkey}JSON serialisation for all values; High-Level Design (HLD) – Per Feature ModuleEach major feature module is designed as a bounded context with its own service, data store access patterns, and external contracts. Services communicate via async events (RabbitMQ) for cross-cutting concerns and via synchronous REST for real-time queries.3.1 Customer Billing ServiceAttributeDetailService NameBillingServiceOwned Tablescustomer_invoice, invoice_line_item, prescription, customerCache Namespacesbill:*, inv:stock (read-only), med:* (read-only), session:*Publishes Eventsinvoice.confirmed, invoice.cancelled, stock.deducted, customer.credit.updatedConsumes Eventsinventory.stock.updated (to refresh inv:stock cache)Sync DependenciesInventoryService (stock check), MedicineService (alternative lookup)External IntegrationsPayment Gateway (Razorpay/PayU), SMS Gateway (MSG91)Billing Flow – Key Design DecisionsStock reservation: when a draft bill is created, quantity_reserved is incremented atomically in Redis (INCRBY with Lua script). This prevents double-selling under concurrent load.FEFO (First Expiry First Out): the billing system automatically selects the batch nearest to expiry first when deducting stock. This is enforced at the InventoryService.deductStock() level.MRP ceiling enforcement: a pre-billing validator asserts sale_price_per_unit &lt;= mrp. Any violation throws a MrpViolationException and the line is rejected.GST computation: CGST = SGST = gst_slab / 2 for intra-state. If customer GSTIN is from a different state, IGST = gst_slab is applied and CGST/SGST = 0.Sequential invoice numbering: INCR on pharmasys:bill:invoice_seq:{branch_id}:{fy} generates the sequence; formatted as PS/{FY}/{5-digit-seq} e.g. PS/25-26/00042.Offline resilience: the PWA stores draft bills in IndexedDB. On reconnect, drafts are synced via POST /invoices/sync endpoint. Conflicts are resolved server-side by timestamp.BILLING FLOW SEQUENCE: 1. FrontDesk scans/enters Rx -&gt; POST /prescriptions (uploads image to S3) 2. FrontDesk searches medicine -&gt; GET /medicines/search?q= (Redis autocomplete) 3. For each medicine: GET /inventory/availability?med={id}&branch={id} (Redis cache) 4. On unavailable: GET /medicines/{id}/alternatives (Redis cache) 5. FrontDesk confirms items -&gt; POST /invoices (draft) -&gt; BillingService reserves stock in Redis (atomic Lua) -&gt; Computes GST per line, applies discounts 6. Payment collected -&gt; PUT /invoices/{id}/confirm -&gt; BillingService CONFIRMS stock deduction (DB write) -&gt; Publishes invoice.confirmed event -&gt; InventoryService consumes -&gt; updates quantity_available in DB -&gt; NotificationService consumes -&gt; sends receipt SMS 7. PDF invoice generated -&gt; stored to S3 -&gt; URL returned to client3.2 Inventory ServiceAttributeDetailService NameInventoryServiceOwned Tablesinventory_stock, expiry_management, damage_log, rackCache Namespacesinv:* (owner), med:* (read-only)Publishes Eventsinventory.stock.received, inventory.stock.updated, inventory.expiry.detected, inventory.reorder.triggeredConsumes Eventsinvoice.confirmed (deduct stock), po.fulfilled (receive stock)Sync DependenciesPurchaseOrderService (validate PO on receipt)Scheduled JobsExpiryDetectionJob (daily 02:00 IST), ReorderAlertJob (every 4 hrs)Inventory Flow – Key Design DecisionsBatch tracking: every stock receipt is stored at batch level (batch_number + expiry_date). Consumption is always traced back to a specific batch for recall scenarios.FEFO selection: InventoryService.GetAvailableBatchesFEFO() returns batches ordered by expiry_date ASC, then received_at ASC as tiebreaker.Reorder trigger: after every stock deduction, if quantity_available &lt;= reorder_threshold, an inventory.reorder.triggered event is published. PurchaseOrderService can auto-draft a PO.Expiry detection job: runs daily, scans inventory_stock WHERE expiry_date &lt;= NOW() + INTERVAL &apos;90 days&apos; AND expiry_alert_sent = false.
